# Bloco 5.2a · Módulo Administração — Aba Usuarios (design)

> Design validado no brainstorming de 2026-07-17. Fonte de regra: Notion EAP 2.6.2
> (Cadastro de Usuários Superadmin/Admin), ADR-06/07, RN-01. Primeiro de dois sub-blocos:
> **5.2a = Usuarios** (este doc); **5.2b = Roles y Permisos** (planejado depois, just-in-time).

## 1. Objetivo e escopo

Entregar a aba **Usuarios** do módulo `/administracion`: CRUD de usuário **staff** (quem autentica com
`type=admin`), com atribuição de role Spatie. O superadmin cria/edita staff; o admin comum apenas
visualiza.

**Dentro do 5.2a:**
- Backend: DTO + Actions + Controller de usuário staff, guard de último superadmin, e uma **rota de
  listagem de roles** (só nome/flag de sistema) para alimentar o select do formulário.
- Frontend: página `/administracion` com a aba Usuarios (tabela + diálogo de cadastro), e ajuste do
  gate de navegação.

**Fora do 5.2a (vai para o 5.2b):**
- Tela de visualização de roles+permissões e criação de role customizada.
- `CreateRoleAction` e o payload de permissões por role.

**Fora de ambos (backlog):**
- Troca de senha pelo próprio usuário e fluxo "esqueci a senha" (não há infra de e-mail).
- Edição/remoção de role customizada.

## 2. Regras de negócio (decididas no brainstorming)

1. **`type` vs role.** `users.type` é enum fechado (`admin|redator|aluno|cliente`) — **não existe
   `superadmin` como type**. Todo staff desta tela nasce `type=admin`; o que separa superadmin de
   admin é a **role Spatie**. A tela nunca grava "superadmin" em `type`.
2. **Escopo da aba.** Usuarios gerencia só `type=admin`. Redator continua no `/personas`;
   cliente/aluno são atores, fora daqui.
3. **Senha no formulário, nasce ativo.** Diferente do ator (inativo, senha placeholder via
   `UserProvisioner`), o staff recebe senha real no create e nasce `is_active=true`. Loga direto.
4. **Role atribuível.** O select de role lista **todas as roles não-`redator`** (superadmin, admin e
   quaisquer customizadas do 5.2b). `redator` nunca é atribuível aqui.
5. **Guard de último superadmin.** Bloquear (422) qualquer operação que deixe o sistema sem nenhum
   superadmin ativo: remover a role superadmin, desativar (`is_active=false`) ou soft-deletar o
   **último** superadmin ativo.
6. **Alcance por perfil.** Admin comum **vê** a lista (gate `identity.user.view`), sem criar/editar.
   Só o superadmin escreve — criar/editar/deletar staff exige `identity.access.manage` (atribuir
   role é ação sensível).

## 3. Backend (domínio Identity)

### 3.1 `UserData` (DTO, `#[TypeScript]`)
- **`fromModel(User $u): self`** (saída) projeta: `id, uuid, name, email, rut, phone, type,
  is_active, roles[]` (`getRoleNames()->all()`).
- **`rules()`** (entrada) para create e update:
  - `name` obrigatório.
  - `email` obrigatório, e-mail, `unique:users,email` com `withTrashed` no check (soft-delete não
    libera o índice — lição 8) e ignorando o próprio id no update.
  - `rut` **opcional**; se presente, `ValidRut` (estrutura) + unicidade via
    `UserProvisioner::ensureRutAvailable` (`withTrashed`, ignora o próprio id).
  - `phone` opcional.
  - `password` obrigatório **só no create** (`min:8`); no update, se ausente mantém a atual, se
    presente re-hash.
  - `role` obrigatório: string, existe em `roles.name`, e `!= 'redator'`.
- `type` **não vem do request** — a Action fixa `admin`.

### 3.2 Actions
- **`CreateStaffUserAction`** — dentro de `DB::transaction`: cria `User` com `type='admin'`,
  `is_active` (default `true`), senha hasheada (cast `hashed` do model), depois `syncRoles([$role])`.
  Reusa `UserProvisioner::ensureRutAvailable` para a checagem de RUT quando houver rut (não usa
  `provision()`, que cria inativo com placeholder — staff é ativo com senha real).
- **`UpdateStaffUserAction`** — `DB::transaction`: atualiza campos; `password` só re-hash se presente;
  `syncRoles([$role])`. **Antes** de aplicar mudança de role/ativação, chama o `SuperadminGuard`.

### 3.3 `SuperadminGuard` (Domain Service, `Domains/Identity/Services/`)
Fonte única da regra de não-lock-out. Método(s) que as Actions/Controller chamam antes de persistir:
- `assertNotLastSuperadmin(User $target)` — lança `ValidationException::withMessages([...])` (422) se
  `$target` é superadmin ativo **e** é o único superadmin ativo restante, para as operações:
  remover a role superadmin, `is_active=false`, ou `delete()`.
- Contagem: `User::role('superadmin')->where('is_active', true)->where('id', '!=', $target->id)->exists()`
  == false ⇒ é o último ⇒ bloqueia.
- Guarda **de aplicação** (não trigger, não evento de model — a mudança de role passa pelo pivot e não
  dispara eventos; mesma razão do `SystemRoleGuard`).

### 3.4 `UserController` (`Domains/Identity/Http/Controllers`)
Controller fino (route-model-binding + Action), sempre retorna `UserData::fromModel($u)`.
- `index` — só `type='admin'` (staff), com roles carregadas. Ordena por nome.
- `show`, `store`, `update`, `destroy` (`destroy` = `$user->delete()`, soft-delete + auditoria —
  nunca query builder, lição 5; passa pelo `SuperadminGuard`).
- **Gates (`HasMiddleware`):** `index`/`show` → `permission:identity.user.view`;
  `store`/`update`/`destroy` → `permission:identity.access.manage`.

### 3.5 `RoleController@index` + `RoleData` (mínimo para o select)
- **`RoleData`** (`#[TypeScript]`) neste bloco: `id, name, is_system` (o `is_system` vem de
  `Role::isSystem()`). O payload de permissões por role fica para o 5.2b (gate `access.manage`).
- **`RoleController@index`** — lista todas as roles (`RoleData::fromModel`), gate
  `permission:identity.user.view` (admin também precisa ver os nomes de role na tabela read-only).

### 3.6 Rotas (`Domains/Identity/routes.php`, sob `auth:sanctum`)
```php
Route::apiResource('users', UserController::class)->only(['index','show','store','update','destroy']);
Route::get('roles', [RoleController::class, 'index']);
```
Gates ficam nos controllers via `HasMiddleware` (padrão do domínio), não na rota.

### 3.7 Tipos gerados
`php artisan typescript:transform` após os DTOs; consumidores TS ajustados **no mesmo commit**
(lição 11).

## 4. Frontend (feature identity + shared)

### 4.1 `shared/api` (ADR-18)
- `usersApi.ts` — `createCrudResource<UserData>('users')`.
- `rolesApi.ts` — cliente de listagem de roles (`GET /roles`) para o select.

### 4.2 Página `/administracion`
- Componente de página com `TabView` (wrapper `shared/ui` se já existir; senão, criar wrapper
  `AppTabView` — decisão do plano). Aba **Usuarios** sempre visível (o gate de rota já garante
  `identity.user.view`). A aba **Roles** entra no 5.2b.
- Substitui o `ModulePlaceholder` atual da rota `/administracion` em `AppRouter.tsx`.

### 4.3 Aba Usuarios
- Molde existente: `useCrudPage` + `CrudDialog` + `useEntityForm` + `AppDataTable`.
- Colunas: nome, e-mail, rut, role(s), estado (ativo).
- Diálogo (form): `name`, `email`, `rut`, `phone`, `password` (só no modo create), `role` (dropdown
  alimentado por `rolesApi`, opções = roles com `name != 'redator'`), `is_active` (switch, default
  ativo). O campo `type` aparece **read-only** exibindo "Admin" — reforça a distinção type/role sem
  permitir edição.
- **Read-only para admin comum:** sem `can('identity.access.manage')`, a tabela não mostra botões de
  criar/editar/excluir (o backend é a autoridade — `can()` é só conveniência de UI, ADR-07).

### 4.4 Navegação
- `NAV_MODULES` (`shared/config/navigation`): trocar o gate de `administracion` de
  `identity.access.manage` → **`identity.user.view`** (admin comum passa a ver o módulo).

## 5. Definition of Done (comportamento provado, não build verde)

Provar contra a API real (curl com `-H 'Origin: <FRONTEND_URL>' -H 'Accept: application/json'`,
lição 12) ou via testes de integração equivalentes:
1. Superadmin cria staff com senha → usuário nasce `type=admin`, `is_active=true`, com a role
   escolhida, e **loga** com a senha definida.
2. Update sem `password` mantém a senha antiga (login segue funcionando); update com `password` troca.
3. RUT duplicado (inclusive de soft-deletado) retorna **422**, não 500.
4. **Guard:** com um único superadmin ativo, tentar (a) remover sua role superadmin, (b) desativá-lo,
   (c) deletá-lo retorna 422. Com dois superadmins, as três operações passam. Cada teste visto
   **reprovando contra o código sem o guard** (lição 10).
5. Admin comum: `GET /api/users` responde 200 (vê a lista); `POST/PUT/DELETE /api/users` responde 403.
6. `role='redator'` no create/update é rejeitado (422).
7. Frontend: superadmin vê botões de CRUD e o fluxo funciona ponta a ponta; admin comum vê a tabela
   sem botões de escrita.

## 6. Testes (integração sqlite `:memory:`, ADR-02)
- `CreateStaffUserAction`: cria ativo com role e senha hasheada; rejeita `redator`; RUT duplicado 422.
- `UpdateStaffUserAction`: preserva senha quando ausente; troca quando presente.
- `SuperadminGuard`: os três caminhos de lock-out (role/desativar/deletar) — reprovando antes do guard.
- `UserController`: gates (admin vê, admin não escreve); `index` só devolve `type=admin`.
