# Bloco 5.2b · Administração — aba Roles y Permisos (design)

> Notion 2.6.3 · ADR-07 · continua o 5.2a (aba Usuarios). Escopo cruza backend+front → **main tree** (P-03).

## Objetivo

Segunda aba do Módulo Administração: visualizar as 3 roles de sistema com suas permissões em
**modo somente-leitura** (ADR-07) e **criar/editar roles customizadas** compostas de um subconjunto
do catálogo fixo de permissões já seedado. Não cria permission nova — só recombina as existentes.

## Regras de negócio (decididas com o João — não re-supor)

1. **Ciclo de vida:** criar + editar role customizada. **Excluir fica fora** (nem backlog por ora).
2. **Guardrail de permissões segregadas:** uma role customizada **não pode** incluir
   `commercial.quote.approve`, `identity.access.manage`, `certification.certificate.revoke` — as três
   exclusivas do superadmin no seeder. Bloqueio no backend (422) e no picker do front (nem aparecem).
3. **Atribuição:** roles customizadas são atribuíveis a staff pela aba Usuarios (5.2a). Já funciona
   sem código novo — `UserData::rules()` valida `role` como `exists:roles,name` + `notIn(['redator'])`,
   e `RoleController@index` devolve todas as roles.
4. **Acesso à tela:** superadmin-only. Leitura+escrita de role sob gate `identity.access.manage`
   (o mesmo que já protege a escrita de usuário). Admin comum não vê a aba.
5. **Sem auditoria de RBAC** neste bloco nem no backlog — decisão explícita do João (proporcional ao
   estágio; `Role` do Spatie não é `Auditable` e não será ligado agora).
6. **Cache:** após qualquer criar/editar de role, `forgetCachedPermissions()` (ADR-07).

## Fundação que o 5.2a já deixou pronta

- `Role` (estende `SpatieRole`): `SYSTEM_ROLES`, `isSystem()`, guards de **delete** e **rename** de
  role de sistema (eventos de model).
- `SystemRoleGuard::assertPermissionsMutable(Role)`: lança `ImmutableSystemRoleException` se a role é
  de sistema — a ser chamado pela `UpdateRoleAction` (o pivot de permissões não dispara evento).
- `RoleController@index` (gate `identity.user.view`) + `RoleData{id,name,is_system}`.
- `RolePermissionSeeder`: catálogo canônico (~36 permissões em 6 domínios) + 3 roles de sistema.

## Arquitetura — backend

### PermissionCatalog (fonte única do catálogo)

Novo `App\Domains\Identity\Support\PermissionCatalog`. Extrai o array `nome → descrição` que hoje
vive `private` no `RolePermissionSeeder`; o seeder passa a consumi-lo (zero mudança de
comportamento — evita descrição duplicada, lição 13).

- `const SEGREGATED = ['commercial.quote.approve', 'identity.access.manage', 'certification.certificate.revoke']`.
- `descriptions(): array<string,string>` — o catálogo cru (fonte que o seeder usa). Métodos
  **estáticos** (catálogo puro, sem estado).
- `toData(): PermissionData[]` — para a API: lista **plana** com `name`, `description`, `group`
  (prefixo antes do 1º `.`), `segregated: bool`. O front agrupa por `group` (achatar evita o mapa
  aninhado que o typescript-transformer não modela bem). `PermissionData` é DTO `#[TypeScript]`.
- `assertAssignable(array $names): void` — lança `ValidationException::withMessages([...])` (422) se
  algum nome for desconhecido **ou** estiver em `SEGREGATED`. Chamada pelas duas Actions (caminho
  programático); a via request também valida em `RoleData::rules()`.

### RoleData (contrato entrada+saída, molde UserData)

Ganha `permissions: string[]`. Saída: `$role->permissions->pluck('name')->all()`. Entrada: o
subconjunto escolhido. `id`/`is_system` viram só-de-saída (`Optional` na entrada).

`rules()`:
- `name` → `required`, `string`, `Rule::notIn(Role::SYSTEM_ROLES)`.
- `permissions` → `array`.
- `permissions.*` → `string`, `exists:permissions,name`, `Rule::notIn(PermissionCatalog::SEGREGATED)`.

**Unicidade do nome fica na Action, não em `rules()`** (molde 5.2a: email/RUT são checados no
`UserProvisioner`, não em `rules()` — assim o update não dá 422 contra o próprio nome). Create:
nome não pode existir. Update: nome não pode existir em **outra** role (ignora o próprio id).

Regenerar `generated.ts` no mesmo commit (lição 11); o campo extra é inócuo para o select do form
de usuário que consome o mesmo `RoleData`.

### Endpoints (`Identity/routes.php`, gate por ação via `HasMiddleware`)

| Método | Rota | Gate | Handler |
|---|---|---|---|
| GET | `/api/roles` | `identity.user.view` | `RoleController@index` (**inalterado**, agora com `permissions[]`) |
| GET | `/api/permissions` | `identity.access.manage` | `PermissionController@index` → `PermissionCatalog::toData()` |
| POST | `/api/roles` | `identity.access.manage` | `RoleController@store` → `CreateRoleAction` |
| PUT | `/api/roles/{role}` | `identity.access.manage` | `RoleController@update` → `UpdateRoleAction` |

`RoleController` passa a implementar as ações de escrita; middleware por ação (index continua no gate
mais brando para o form de usuário; store/update no gate sensível).

### Actions (`DB::transaction` + `forgetCachedPermissions()` no fim)

- **CreateRoleAction**: checa nome disponível (não existe em `roles`, 422 se colide);
  `PermissionCatalog::assertAssignable($data->permissions)`; cria `Role` (`guard_name=web`);
  `syncPermissions($data->permissions)`; retorna `$role->load('permissions')`.
- **UpdateRoleAction**: `SystemRoleGuard::assertPermissionsMutable($role)` (**403** se sistema —
  `ImmutableSystemRoleException` é `HttpException(403)`, não validação);
  checa nome disponível ignorando o próprio id (422 se colide com outra role);
  `assertAssignable`; renomeia se mudou; `syncPermissions`; retorna `$role->load('permissions')`.
  Rename/edit de role de sistema barrado pelo guard/model.

## Arquitetura — frontend

- **`AdministracionPage`**: adiciona 2ª `ModuleTab header={t('admin.tabRoles')}` com a `RolesTable`
  dentro (sem componente `Panel` intermediário). Aba só renderiza para `can('identity.access.manage')`.
- **`shared/api/rolesApi`**: `createCrudResource<RoleData>('roles')` (ADR-18: cliente REST no
  shared). **`features/identity/api/usePermissionCatalog.ts`**: hook TanStack read-only do catálogo.
- **`components/Admin/`**:
  - `RolesTable` — lista roles, badge sistema/custom, ação ver/editar.
  - `RoleDialog` — molde `CrudDialog`, view=edit=create. Role de sistema: nome e permissões
    read-only (mostra todas, inclusive segregadas). Role custom: nome editável + picker de permissões
    (checkbox por permissão, **agrupado por domínio**, com a descrição como label; permissões
    segregadas **não entram no picker**).
  - Hooks `useRolesPage`/`useRoleForm` (molde `useUsersPage`/`useStaffUserForm`).
- **i18n**: 3 locales com chaves idênticas (`admin.tabRoles`, `role.*`), es-CL como referência.
- `can()` é conveniência de UI; a autorização real é dos gates da API (ADR-07).

## Definition of done — prova e2e contra a API real

Via curl (Origin + Accept, lição 12), logado como superadmin salvo onde indicado:

1. `POST /api/roles` `{name:'coordinador', permissions:[subconjunto NÃO-segregado]}` → **201**, role
   criada com exatamente essas permissões.
2. `POST /api/roles` com uma permissão segregada (`identity.access.manage`) → **422**.
3. `POST /api/roles` `{name:'admin', ...}` (nome de sistema) → **422**.
4. `PUT /api/roles/{id da custom}` trocando permissões → **200**, permissões novas aplicadas.
5. `PUT /api/roles/{id da role de sistema}` → **403** (`SystemRoleGuard` → `ImmutableSystemRoleException`,
   que é `HttpException(403)`; distinto do 422 de validação).
6. `GET /api/permissions` logado como **admin comum** (sem `identity.access.manage`) → **403**.
7. `POST /api/users` `{..., role:'coordinador'}` → **201**; o staff criado **loga** (a role custom é
   atribuível e o cache foi invalidado).

Suíte verde e build verde **não** fecham o bloco — só a prova acima fecha.

## Fora de escopo (não construir)

- Excluir role customizada (nem backlog).
- Criar/editar/remover permissões (o catálogo é fixo).
- Auditoria de RBAC.
- Data-scoping por role (ex.: "redator só vê as suas turmas") — é tarefa de Policies, outro bloco.
